Ensure keyboard accessibility

Keyboard accessibility is a fundamental part of enabling various input methods, including switches, speech input, and screen readers.

How to test with a keyboard:

  • tab key to progress through links and controls
  • shift-tab to reverse
  • enter to follow links
  • space to select form controls (e.g. tick boxes)
  • arrows for radio-buttons / tabs.

Test zooming and text sizing

People use their browser to change the size of content, and people with low vision may use that to a more extreme level, the guidelines specify a level of 320 (CSS) px wide. That is equivalent to 400% for a 1280px wide browser.

Users can also over-ride the font-family and spacing for text, which can break a fragile layout. The guidelines specify some values to test with, and we use a bookmarklet to do so.

NHS UK must use a responsive design which helps with zooming, and take care with layouts that limit text-expansion.

Setting page titles

Each page title should be unique and descriptive, so for non-content managed pages those titles need to be defined. For content managed pages the template should help/enforce the agreed pattern.

Best practice is generally: Page name | Section – Sitename

For example: Achalasia | Health A-Z - NHS

The main tittle of the page should match the start of the page title.

Setting the (human) language of the content

In order for screenreaders (and search engines) to know what language the page is a small bit of markup is needed in each page.

In the HTML element the language can be set to English for the whole page: [add code component] html lang="en"

Where there are changes of language within the page, these should be marked up as well:

[Add code component] You can search for health professionals for the area you are staying in via l’Assurance Maladie website (information in French only).

The lang attribute can be applied to any container.
