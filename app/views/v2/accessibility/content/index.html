Understandable content

Making the text content as easy to understand as possible is the most important aspect of the website, it will help more audiences than any other accessibility requirement.

NHS services should follow the Content style guide part of the service manual.

Alternative text for images

For people who cannot see an image, having an alternative is vital to understand the content.

In web pages "alt text" is required for each image. The content of the alt text depends on the image type and context. For images that are part of the main content area of a page they should describe the image in a way that is meaningful in context.

Imagine you were reading the page out to a friend; how would you describe the image?

{{ image({ "src": "https://via.placeholder.com/300x150", "alt": "Placeholder image", "caption": "Caption" }) }}

The caption underneath the image can assume you understand the image, either by seeing it or reading the alt text.

Headings

Headings provide descriptions of content, and for some people they provide an effective method of navigating the page.

The main heading of the page is added by the [title field in wagtail].

Each section of the page should start with a #H2, and each sub-section should start with a #H3. It is possible to provide sub-sub-sections with a #H4.

The question for each heading is whether it is a sub-section of the previous heading. If not, it should be at the same level as (or higher than) the previous section.

The size and style of text is not the deciding factor about heading level, the structure of the page is the key factor.

For more examples, see the [headings pattern].

Data tables

Tables should only be used if there is a relationship between the "header" cells, and the "data" cells.

Assistive technology makes use of the relationship to announce the header before each cell and provide column and row information.

{{ image({ "src": "https://via.placeholder.com/300x150", "alt": "Placeholder image", "caption": "Caption" }) }}

In the example data table, the last row would be read as: "Skin symptoms: Itchy blisters. Possible cause: shingles, chickenpox.", the headers and data are read out together.

It is important to only use tables where this type of relationship is required, otherwise it leads to confusing content.

Colour and location

For people who cannot see a page or have converted it into a different format, the colour and location of page elements will change. Therefore providing instructions that rely on those aspects will be problematic.

For example, do not say "Press the red button on the right".

  • For someone who is zoomed in, the button may not be on the right
  • For someone with colour blindness, the button may not be differentiated from a green button
  • For someone using a screenreader, the colour is not available and the location is simple up or down

Defining link and control names

Links needs to make sense out of context as some people experience them that way, so each link on a page should clearly describe where it will take you if you choose to follow it.

{{ image({ "src": "https://via.placeholder.com/300x150", "alt": "Placeholder image", "caption": "Caption" }) }}

Ideally link text should match the heading of the target page. So if the target page of the link has a heading “Sleep and tiredness”, that would make good link text.

Also:

  • - Avoid ambiguous phrases such as “click here”, “read more” and “find out more”
  • If links open in a new window this information should be included in the link phrase (e.g. “Link name (opens in new window)”)

Setting page titles

Each page title should be unique, descriptive and concise. It is the first thing read out to a screenreader when landing on a page, and the link shown in search results. In both cases a good page title helps when finding the page or switching between tabs.

For content managed pages the title is based on [the title field in Wagtail], and fits into a templated pattern.
